how to use git with others in github

`——
首先先解决几个问题？

  • git是什么？
    Git是一个开源的分布式版本控制系统，可以有效、高速的处理从很小到非常大的项目版本管理。Git 是 Linus Torvalds 为了帮助管理 Linux 内核开发而开发的一个开放源码的版本控制软件

  • git可以做什么?
    1、从服务器上克隆数据库（包括代码和版本信息）到单机上。
    2、在自己的机器上创建分支，修改代码。
    3、在单机上自己创建的分支上提交代码。
    4、在单机上合并分支。
    5、新建一个分支，把服务器上最新版的代码fetch下来，然后跟自己的主分支合并
    6、生成补丁（patch），把补丁发送给主开发者。
    7、看主开发者的反馈，如果主开发者发现两个一般开发者之间有冲突（他们之间可以合作解决的冲突），就会要求他们先解决冲突，然后再由其中一个人提交。如果主开发者可以自己解决，或者没有冲突，就通过。
    8、一般开发者之间解决冲突的方法，开发者之间可以使用pull 命令解决冲突，解决完冲突之后再向主开发者提交补丁。

  1. 由于本片文章重点不在于git的配置,关于如何安装配置git，以及将github作为git的中心库的过程可以自行百度。[git配置过程][1]

下面的操作是在上诉过程配置完成后进行的，我们通过解决实际开发中所遇到的问题对git进行一一的讲解(我们以命令行的方式进行讲解)。
1.如何将我们本地的代码同步的github？
首先先熟悉2个概念(本地仓库，中心仓库)
本地仓库:我们放置我们代码的文件夹
中心仓库:github的仓库(相当于svn的中心服务器)。
代码同步的过程实际上是将本地仓库的代码同步到github中心仓库。
1 新建本地仓库。
右键想作为本地仓库的文件夹，选择git bash(如果成功安装git后会出想这个选项)。
输入命令:
git init（将该文件夹初始化为本地仓库）
2 当我们修改了代码,首先需提交到本地仓库
git status (查看当前的文件的改动)
git add +文件名/×（×代表所有文件）
git commit -m ‘备注信息’
上面2个步骤就是将修改的代码同步到本地仓库
git remote add origin git://github.com/zjccnu/test.git（为你的github仓库地址）
git push 将本地代码上传的github仓库

  1. 多个人合作时怎么向别人的项目提交代码。
    比如 A B C三人合作开发一个项目(A为主导者即A创建项目)
    1. A创建项目test(姑且叫这个名字)
    2. B C去fork A的test项目
    3. B C操作一样,下面说B的操作
    4. B fork后自己的仓库将会也有一个test项目，B将该项目clone到本地
      命令为:
      git clone +’B的test项目github地址’
      5.B在本地进行自己的编码工作。
      下面提到一个新概念:分支 branch
      什么是分支以及为什么要分支？
      分支相当于对我们clone的test项目的一个浅拷贝。我们在分支上的修改将不会影响到我们的主分支
      建立dev分支并切换
      git checkout -b dev
      6.下面B的所有工作将在dev分支上进行，当B改完后，需要将该分支合并到主分支，然后同步到自己的远程仓库
      合并分支:B首先切换到master分支
      git merge dev
      将dev分支合并到master分支上。
      将修改同步到本地仓库，以及同步到自己github上的命令同上。
      B在去自己的github的test项目里点击pull request 提交请求

3: 当B C同时修改后A合并了，BC怎么保证自己的远程仓库和A一致
思路如下(B从本地获取A的代码，然后将A的代码同步到自己的github从而间接的实现了同步)
BC 一样故下面以B为例进行分析
首先新建分支 :git checkout -b dev
git remote -v（查看当前源）
B在test的本地仓库添加upstream为A的远程仓库地址
命令如下 ：
git remote add upstream +A的远程仓库地址
获取A的test项目
git fetch upstream
git merge upstream/master
（这里的master只的是从A远程获取的master分支,不是自己本地的master分支）
再将dev分支和本地的master分支合并
(
git checkout master(切换到master分支)
git merge dev （将dev合并到master）
git branch -D dev 上出dev分支 )
git push（同步到B的github仓库）
）

vue.js打包问题总结

  1. vue 打包静态界面报错
    可能原因：index.html 中 和compnetnts中进行了重复的引用
    解决方案: 去掉重复的引用
  2. vue 打包es6语法报错问题(dev下不错但是build报错)
    可能原因:1.babel配置问题，具体配置自行百度
    2.webpack.prod.conf.js 在配置发布配置时默认采用
     new webpack.optimize.UglifyJsPlugin({
         compress: {
           warnings: false
         },
         sourceMap: true
       })
  不会将es6转es5 直接去掉这个代码，会采用babel进行编辑
    3:vue打包后静态资源找不到，在config 中的index.js里，我们看到如下配置
    index: path.resolve(__dirname, '../dist/index.html'),
 assetsRoot: path.resolve(__dirname, '../dist'),
 assetsSubDirectory: 'static',
 assetsPublicPath: '/',
 productionSourceMap: true,
    静态资源默认为与src同级的static下，所以需要自己在src同级目录下建立static，并将自己的静态资源放入即可

4.部署到apache问题，将apache的默认目录改为打包后的文件目录,即打包后的文件在dist下，我们把apache2的默认目录设置为dist即可

什么是 Cmd Markdown

您可以使用很多工具书写 Markdown，但是 Cmd Markdown 是这个星球上我们已知的、最好的 Markdown 工具——没有之一 ：）因为深信文字的力量，所以我们和你一样，对流畅书写，分享思想和知识，以及阅读体验有极致的追求，我们把对于这些诉求的回应整合在 Cmd Markdown，并且一次，两次，三次，乃至无数次地提升这个工具的体验，最终将它演化成一个 编辑/发布/阅读 Markdown 的在线平台——您可以在任何地方，任何系统/设备上管理这里的文字。

1. 实时同步预览

我们将 Cmd Markdown 的主界面一分为二，左边为编辑区，右边为预览区，在编辑区的操作会实时地渲染到预览区方便查看最终的版面效果，并且如果你在其中一个区拖动滚动条，我们有一个巧妙的算法把另一个区的滚动条同步到等价的位置，超酷！

2. 编辑工具栏

也许您还是一个 Markdown 语法的新手，在您完全熟悉它之前，我们在 编辑区 的顶部放置了一个如下图所示的工具栏，您可以使用鼠标在工具栏上调整格式，不过我们仍旧鼓励你使用键盘标记格式，提高书写的流畅度。

3. 编辑模式

完全心无旁骛的方式编辑文字：点击 编辑工具栏 最右侧的拉伸按钮或者按下 Ctrl + M，将 Cmd Markdown 切换到独立的编辑模式，这是一个极度简洁的写作环境，所有可能会引起分心的元素都已经被挪除，超清爽！

4. 实时的云端文稿

为了保障数据安全，Cmd Markdown 会将您每一次击键的内容保存至云端，同时在 编辑工具栏 的最右侧提示 已保存 的字样。无需担心浏览器崩溃，机器掉电或者地震，海啸——在编辑的过程中随时关闭浏览器或者机器，下一次回到 Cmd Markdown 的时候继续写作。

5. 离线模式

在网络环境不稳定的情况下记录文字一样很安全！在您写作的时候，如果电脑突然失去网络连接，Cmd Markdown 会智能切换至离线模式，将您后续键入的文字保存在本地，直到网络恢复再将他们传送至云端，即使在网络恢复前关闭浏览器或者电脑，一样没有问题，等到下次开启 Cmd Markdown 的时候，她会提醒您将离线保存的文字传送至云端。简而言之，我们尽最大的努力保障您文字的安全。
